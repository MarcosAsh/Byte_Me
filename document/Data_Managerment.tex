\section{Introduction}
This document outlines how data is structured, stored, managed, validated, and used within the Byte Me platform. The system connects food sellers with organisations to reduce food waste through discounted surplus food bundles. 
The database is implemented using PostgreSQL and follows a relational schema with strong integrity constraints, automated triggers, and forecasting capabilities.

\section{Data Structure}
The system is built using a relational database consisting of several core entity groups:

\subsection{Accounts}
\begin{itemize}
\item \textbf{user\_account}: Stores authentication data and roles (SELLER, ORG\_ADMIN)
\item \textbf{seller}: Business users offering food bundles
\item \textbf{organisation}: Organisations collecting food
\item \textbf{organisation\_streak\_cache}: Tracks engagement metrics
\end{itemize}

\subsection{Reference Data}
\begin{itemize}
\item \textbf{category}: Food types (e.g., Bakery, Hot Meals)
\item \textbf{pickup\_window}: Time slots for collection
\item \textbf{badge / organisation\_badge}: Gamification and achievements
\end{itemize}

\subsection{Core Business Data}
\begin{itemize}
\item \textbf{bundle\_posting}: Listings of surplus food bundles
\item \textbf{reservation}: Booking of bundles by organisations
\item \textbf{reservation\_status\_history}: Tracks status changes over time
\item \textbf{rescue\_event}: Completed food collections
\item \textbf{issue\_report}: Problems reported by organisations
\end{itemize}

\subsection{Forecasting and Analytics}
\begin{itemize}
\item \textbf{demand\_observation}: Historical demand data
\item \textbf{forecast\_run}: Model training runs
\item \textbf{forecast\_output}: Predictions for postings
\item \textbf{seller\_metrics\_weekly}: Aggregated performance metrics
\item \textbf{forecast\_action}: Seller decisions based on forecasts
\end{itemize}

\section{Data Lifecycle}
Data flows through the system in the following stages:
\begin{enumerate}
\item \textbf{Creation}
\begin{itemize}
\item Sellers create bundle postings
\item Organisations create reservations
\end{itemize}
\item \textbf{Processing}
\begin{itemize}
\item Reservations move through statuses: RESERVED, COLLECTED, CANCELLED, EXPIRED, NO\_SHOW
\item Status changes are recorded in \texttt{reservation\_status\_history}
\end{itemize}
\item \textbf{Outcome Recording}
\begin{itemize}
\item Completed collections create \texttt{rescue\_event} records
\item Issues can be logged in \texttt{issue\_report}
\end{itemize}
\item \textbf{Analytics and Forecasting}
\begin{itemize}
\item Historical data stored in \texttt{demand\_observation}
\item Models trained and stored in \texttt{forecast\_run}
\item Predictions stored in \texttt{forecast\_output}
\end{itemize}
\item \textbf{Action and Feedback}
\begin{itemize}
\item Sellers respond to predictions (\texttt{forecast\_action})
\item Outcomes measured in metrics tables
\end{itemize}
\item \textbf{Archival}
\begin{itemize}
\item Data is retained for analytics and reporting
\item No automatic deletion is currently defined
\end{itemize}
\end{enumerate}

\section{Data Integrity and Quality}
Data quality is enforced through:

\subsection{Constraints}
\begin{itemize}
\item Quantity checks such that $quantity\_reserved \leq quantity\_total$
\item Valid ranges (e.g., discount between $0$--$100$)
\item Time validation such that $pickup\_end > pickup\_start$
\item Non-negative values for price and weight
\end{itemize}

\subsection{Foreign Keys}
Ensure relationships between entities (e.g., reservations must reference postings).

\subsection{Triggers}
Automated rules include:
\begin{itemize}
\item Role validation for sellers and organisations
\item Prevention of overselling bundles
\item Automatic adjustment of reserved quantities
\item Enforcement that rescue events require collected reservations
\end{itemize}

\subsection{Validation via Seed Data}
Seed scripts intentionally insert invalid data to confirm constraints and business rules are enforced.

\section{Data Storage and Security}
\subsection{Storage}
\begin{itemize}
\item Hosted on PostgreSQL (Railway platform)
\item Uses UUIDs for all primary keys
\item Data stored in normalized relational tables
\end{itemize}

\subsection{Security Measures}
\begin{itemize}
\item Passwords stored as bcrypt hashes
\item Role-based access control via user roles
\item Sensitive fields (e.g., claim codes) stored as hashes
\end{itemize}

\subsection{Access Control}
\begin{itemize}
\item SELLER: manages postings
\item ORG\_ADMIN: manages reservations and collections
\item Database-level constraints enforce role usage
\end{itemize}

\section{Data Organisation}
\subsection{Naming Conventions}
\begin{itemize}
\item Snake\_case for tables and columns
\item Descriptive naming conventions
\end{itemize}

\subsection{Versioning}
\begin{itemize}
\item Managed using Flyway migrations
\item Schema and seed scripts are version-controlled
\end{itemize}

\subsection{Indexing}
Indexes improve performance for:
\begin{itemize}
\item Searching active postings
\item Retrieving reservations by organisation
\item Forecast queries
\item Seller activity tracking
\end{itemize}

\section{Data Sharing and Access}
Data is accessed internally by application services. No external public data sharing is currently defined. Future enhancements may include analytics dashboards and sustainability reporting.

\section{Forecasting and Decision Support}
The system includes a predictive analytics pipeline:
\begin{enumerate}
\item Historical demand captured in \texttt{demand\_observation}
\item Models trained and stored in \texttt{forecast\_run}
\item Predictions generated in \texttt{forecast\_output}
\item Sellers act on predictions (\texttt{forecast\_action})
\item Outcomes measured in reservations and metrics
\end{enumerate}
This enables demand forecasting, waste reduction optimisation, and continuous model improvement.

\section{Testing and Reproducibility}
\subsection{Seed Data}
\begin{itemize}
\item Repeatable migration script
\item Uses \texttt{ON CONFLICT DO NOTHING} for idempotency
\end{itemize}

\subsection{Test Coverage}
\begin{itemize}
\item Includes valid and invalid data scenarios
\item Verifies constraints, triggers, and business rules
\end{itemize}

\subsection{Scalability Testing}
Bulk data generation simulates real-world usage with large volumes of data.

\section{Backup and Recovery}
Recommended practices:
\begin{itemize}
\item Regular automated database backups
\item Point-in-time recovery support
\item Secure storage of backup data
\end{itemize}

\section{Ethical and Legal Considerations}
\begin{itemize}
\item Minimal personal data stored (email only)
\item Passwords securely hashed
\item Supports transparency via issue reporting
\item GDPR considerations include data minimisation and potential deletion on request
\end{itemize}

\section{Conclusion}
The Byte Me data system demonstrates strong relational database design, robust validation mechanisms, and a complete data lifecycle. It integrates forecasting and analytics to support data-driven decision making, ensuring reliability, scalability, and real-world applicability.
